# Making an SSH client the hard way
**Today, we’re launching a web-based SSH client: Tailscale SSH Console.**
From the Tailscale admin console, admins will now see a little “SSH…” button to connect to devices running [Tailscale SSH](/kb/1193/tailscale-ssh/). Click this, and you’ll pop open an SSH client, right in your browser. Tailscale SSH Console is now available in beta.
To start a Tailscale SSH Console session, click “SSH” on the device, select the username you want to connect as, and reauthenticate.
Web-based SSH clients aren’t new. Nearly every VPS and cloud provider already lets you connect to your VMs from the web — so how is this different? With Tailscale SSH Console, your browser *becomes* a Tailscale client, and joins your [tailnet](/kb/1136/tailnet/) in the same way as [any other device that you run Tailscale on](/blog/how-tailscale-works/). To make this possible, we ported the following to [WebAssembly](https://webassembly.org/): the Tailscale client, [WireGuard®](https://www.wireguard.com/), a complete [userspace network stack](https://gvisor.dev/docs/user_guide/networking/) (from [gVisor](https://gvisor.dev/)), and an SSH client.
When you click the “SSH…” button, we create an [ephemeral](/kb/1111/ephemeral-nodes/) [auth key](/kb/1085/auth-keys/) in the Tailscale coordination server, and then give that auth key to a Tailscale client loaded by your browser, which starts a node with that key. The client then creates an in-memory-only WireGuard keypair and starts to communicate with the Tailscale coordination server (to discover the rest of your tailnet) and [DERP relays](/blog/how-tailscale-works/#encrypted-tcp-relays-derp) (to be able to connect to other nodes). Like other clients that use DERP, the traffic that passes through the relays is encrypted and not visible to Tailscale.
And because this uses Tailscale SSH, which authenticates each packet based on its [WireGuard cryptographic identity](/kb/1193/tailscale-ssh/#how-does-it-work), we don’t need to ask you for a password, or have you upload a public key, or have you manage an `authorized\_keys` file. Just click “SSH…” *and you’re in* 🪄.
### One year earlier
Tailscale SSH Console started out as [Brad’s vacation hack](https://twitter.com/bradfitz/status/1450916922288005122) to see if [Go’s WebAssembly support](https://github.com/golang/go/wiki/WebAssembly) was good enough to allow the Tailscale client to be compiled for the browser. After some [hacking](https://github.com/tailscale/tailscale/pull/3155) [and](https://github.com/tailscale/tailscale/pull/3195) [slashing](https://github.com/tailscale/tailscale/pull/4709) to fix build errors (including changes [to wireguard-go](https://git.zx2c4.com/wireguard-go/commit/?id=82e0b734e5d28f96205de65fd2179e08fd91b1db)), we had a binary that would load in the browser — but it couldn’t connect to anything.
Tailscale SSH Console connects your browser to another node running Tailscale SSH.
Tailscale tries really hard to establish connections, and if a direct path is not available, it will use [a DERP relay server](/blog/how-tailscale-works/#encrypted-tcp-relays-derp) to forward packets. However, a browser is an even more inhospitable environment than normal, and the DERP HTTP client [reaching into connection internals](https://github.com/tailscale/tailscale/blob/7cf8ec8108f4e26092bf07f698c8c5ce70d27d13/derp/derphttp/derphttp_client.go#L231-L362) was not compatible with the [`fetch()`-based HTTP transport](https://github.com/golang/go/blob/1f068f0dc7bc997446a7aac44cfc70746ad918e0/src/net/http/roundtrip_js.go) that ends up being used by Go when targeting WebAssembly. Like all problems in computer science, it could be solved with a layer of indirection. We added WebSockets [as another transport](https://github.com/tailscale/tailscale/pull/3165) for DERP, and then wrapped a WebSocket connection into something that’s close enough to a `net.Conn` for the rest of the DERP code to use with no modifications. We were lucky that the [Go WebSockets library](https://github.com/nhooyr/websocket) that we use (and [sponsor](https://github.com/orgs/tailscale/sponsoring)) already had [good WebAssembly support](https://github.com/nhooyr/websocket/pull/142), and only needed [a small tweak](https://github.com/nhooyr/websocket/pull/350). We would later use this same [“tunnel it through WebSockets” approach](https://github.com/tailscale/tailscale/pull/4781) for the [coordination server](/blog/how-tailscale-works/#the-control-plane-key-exchange-and-coordination) protocol.
Though we had a working WebAssembly binary, it was rather…large. While this is [unavoidable](https://github.com/golang/go/wiki/WebAssembly#reducing-the-size-of-wasm-files) to some degree when building Go for WebAssembly, there was surely some low-hanging fruit. A friend had recently released [Weave](https://github.com/evmar/weave), a `.wasm` viewer, and we [extended it with a tree view](https://github.com/evmar/weave/pull/5) to make it easier to see which packages and files were contributing to the binary size the most. We found a few [unnecessary](https://github.com/tailscale/tailscale/pull/4802) [dependencies](https://github.com/tailscale/tailscale/pull/4813), and together with [some build settings tweaks](https://github.com/tailscale/tailscale/blob/9d04ffc7823dbd5fb18d5e5e9dea462846128969/cmd/tsconnect/common.go#L197-L207) we were able to shave off a few megabytes. However, there’s still [more work](https://github.com/tailscale/tailscale/issues/5142) to be done here.
## Building a product
Just being able to have a Tailscale client run in the browser is cool in and of itself, but there was still the question of “OK, but what would you use it for?”. This occurred while we were developing [Tailscale SSH](/kb/1193/tailscale-ssh/), and we saw a natural application — using this new capability to create a secure, in-browser, SSH client. We could have a client that let you browse through the devices on your tailnet that were accessible to you, made use of the [policy to let you pick a username](/kb/1193/tailscale-ssh/#ensure-tailscale-ssh-is-permitted-in-acls) on the host, and integrated with [check mode](/kb/1193/tailscale-ssh/#configure-tailscale-ssh-with-check-mode).
The most natural place for this SSH client would be Tailscale’s [admin console](https://login.tailscale.com/admin), but that is part of the [closed source coordination server](/blog/opensource/#why-our-coordination-server-is-closed-source). We liked the idea of the WebAssembly client being open source, but we needed a way to reuse it in our closed source repo. We ended up making [an NPM package](https://www.npmjs.com/package/@tailscale/connect) to allow this, which was a bit of a novelty in our Go-centric world. Though this package was developed for internal use, we want to support [other interesting applications](https://leaningtech.com/webvm-virtual-machine-with-networking-via-tailscale/) — so please [get in touch](https://github.com/tailscale/tailscale/issues/new/choose) if you have something in mind.
Adding the ability to create SSH sessions from our admin console led to some interesting user experience discussions. Would these sessions be transient, just open for a few minutes to check on something, or longer affairs? Would the user want to have multiple sessions running concurrently? As we pondered adding tabs, and possibly making things draggable and minimizable, it became apparent that we were effectively creating [a bug-ridden, informally-specified](https://en.wikipedia.org/wiki/Greenspun's_tenth_rule) implementation of half a window manager.
Rather than get into that business, we decided we should use real windows, and thus allow the user to have the same flexibility that they would with a real terminal. To make these windows fast, they are “child” windows that the main admin console opens and uses as rendering surfaces – all of the logic and state still remains in the parent window. This approach has good performance characteristics, but did require [some](https://github.com/xtermjs/xterm.js/pull/4069) [changes](https://github.com/radix-ui/primitives/pull/1677) to our frontend dependencies. To make it easier to iterate on these interface changes, we recreated [some tooling](https://blog.persistent.info/2016/01/static-resource-grafting.html) to allow us to use in-development code with our prod tailnet, so that we could try out more realistic scenarios.
Maya demonstrates how to use Tailscale SSH Console.
To try out Tailscale SSH Console: first, [enable Tailscale SSH](/kb/1193/tailscale-ssh/#configure-tailscale-ssh) on the device you’re trying to connect to. Then, from the [machines tab](https://login.tailscale.com/admin/machines) of the admin console, select “SSH…” to connect to any machine that your access controls allow you to connect to. Re-authenticate, and you’re in. [See the documentation](/kb/1216/tailscale-ssh-console/) to learn more.
